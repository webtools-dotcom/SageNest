# SageNest — New Tool Addition Guide

This document is the complete, self-contained reference for adding any new interactive tool to SageNest Health. Read it fully before writing a single line of code. Following it exactly means zero broken routes, zero broken builds, a UI that is pixel-consistent with every existing tool, and pages structured to rank for the maximum number of keywords.

---

## 0. What You Are Building

A SageNest tool is:
- A **pure client-side React page** — zero backend, zero API calls, all logic runs in the browser
- A **calculation lib file** containing pure TypeScript functions (no React, no imports from React)
- Registered in **four places**: `tools.ts`, `App.tsx`, `_redirects`, and its own page file

It is NOT:
- A blog post (different pipeline entirely — see `SAGENEST BLOG MASTERGUIDE.md`)
- A page that calls any external API at runtime (CSP blocks it anyway)
- A page that stores user data anywhere (privacy guarantee of the site)


## 2. Step-by-Step Process

### Step 1 — Create the calc lib: `src/lib/<toolSlug>Calc.ts`

This file must be **pure TypeScript with zero React**. It exports:
1. Result interface(s)
2. The main calculation function
3. A validation function

Pattern to follow exactly:

```typescript
// src/lib/exampleCalc.ts
import { addDays, normalizeDate } from './calc';  // use these helpers when working with dates

// Always export your result shape as an interface
export interface ExampleResult {
  someDate: Date;
  someNumber: number;
  someLabel: string;
}

// Always export a validation result interface
export interface ValidationResult {
  valid: boolean;
  errors: string[];
}

// Main calculation — pure function, no side effects
export const calculateExample = (inputDate: Date, inputNumber: number): ExampleResult => {
  const normalized = normalizeDate(inputDate);
  return {
    someDate: addDays(normalized, inputNumber),
    someNumber: inputNumber * 2,
    someLabel: 'Example label',
  };
};

// Validation — always call this before running calculation
export const validateExampleInputs = (inputDate: Date, inputNumber: number): ValidationResult => {
  const errors: string[] = [];

  if (!(inputDate instanceof Date) || Number.isNaN(inputDate.getTime())) {
    errors.push('Please enter a valid date.');
  }

  if (!Number.isFinite(inputNumber) || inputNumber < 1 || inputNumber > 100) {
    errors.push('Number must be between 1 and 100.');
  }

  return { valid: errors.length === 0, errors };
};
```

**Rules for calc files:**
- Import only from `'./calc'` or `'./dateHelpers'` — never from React or any page
- `normalizeDate()` strips time from a Date to midnight local — always use it before date math
- `addDays(date, n)` adds n days — always use this, never manually do `date.getTime() + n * 86400000`
- All error strings must be human-readable sentences (they display directly to the user)
- Validation error strings: mention the field they relate to so the page can `.find()` them by keyword

## 3. SEOHead — Exact Usage Pattern

```tsx
const jsonLd = useMemo(
  () => [
    {
      '@context': 'https://schema.org',
      '@type': 'WebApplication',
      name: 'Tool Name',
      url: 'https://sagenesthealth.com/tool-slug',
      applicationCategory: 'HealthApplication',
      operatingSystem: 'Any',
      description: 'Same as meta description.',
    },
    {
      '@context': 'https://schema.org',
      '@type': 'FAQPage',
      mainEntity: FAQ_ITEMS.map((item) => ({
        '@type': 'Question',
        name: item.question,
        acceptedAnswer: { '@type': 'Answer', text: item.answer },
      })),
    },
  ],
  [],
);
```

Always include both `WebApplication` and `FAQPage` schemas. Never pass `noIndex={true}` for a real tool page.

## 8. Form Validation Pattern

```tsx
const [hasSubmitted, setHasSubmitted] = useState(false);
const [touched, setTouched] = useState({ fieldOne: false, fieldTwo: false });

const errors = useMemo<FieldErrors>(() => {
  const next: FieldErrors = {};
  if (!fieldOne) next.fieldOne = 'Enter your field one value.';
  if (!next.fieldOne && !next.fieldTwo) {
    const validation = validateExampleInputs(new Date(fieldOne), Number(fieldTwo));
    const f1Err = validation.errors.find((e) => e.toLowerCase().includes('keyword'));
    if (f1Err) next.fieldOne = f1Err;
  }
  return next;
}, [fieldOne, fieldTwo]);

const showFieldOneError = Boolean(errors.fieldOne) && (touched.fieldOne || hasSubmitted);

const result = useMemo(() => {
  if (errors.fieldOne || errors.fieldTwo || !fieldOne || !fieldTwo) return null;
  return calculateExample(new Date(fieldOne), Number(fieldTwo));
}, [errors.fieldOne, errors.fieldTwo, fieldOne, fieldTwo]);

const onSubmit = (event: FormEvent<HTMLFormElement>) => {
  event.preventDefault();
  setHasSubmitted(true);
  setTouched({ fieldOne: true, fieldTwo: true });
};
```

Rules: `errors` is always `useMemo` not `useState`. `result` returns `null` if any error exists. Form uses `noValidate`.

## 9. The `toInputDate` Helper

Copy this into every page file that has a date input:

```tsx
const toInputDate = (value: Date): string => {
  const year = value.getFullYear();
  const month = `${value.getMonth() + 1}`.padStart(2, '0');
  const day = `${value.getDate()}`.padStart(2, '0');
  return `${year}-${month}-${day}`;
};
```

## 10. SEO Strategy — Maximize Keyword Coverage

This section is mandatory. Every tool must be built with SEO as a first-class concern.

### 10.1 — The four keyword tiers every tool must target

**Tier 1 — Primary keyword** (the exact phrase people search to find this tool)
- Appears in: URL slug, H1, title tag, meta description, first sentence of hero description
- One primary keyword per tool only. Don't stuff the H1.
- Example: `"when does morning sickness end"`

**Tier 2 — Secondary keywords** (close variants, action-intent versions)
- Appears in: first H2 in educational section, trust badge, calculator-subtitle
- 2–3 secondary keywords woven naturally into above-the-fold content
- Example: `"morning sickness end date"`, `"when does nausea stop in pregnancy"`

**Tier 3 — Long-tail keywords** (specific questions people type into Google)
- Appears in: FAQ question text — each FAQ question IS a long-tail keyword
- Minimum 5 FAQ items, each written as a real complete-sentence search query
- Example: `"Does morning sickness end at exactly 12 weeks?"`, `"Can morning sickness last all 9 months?"`

**Tier 4 — LSI / semantic keywords** (related terms Google uses to confirm topic relevance)
- Appears in: educational section body text, result card labels, info boxes
- Write naturally about the full topic and these appear organically
- Example for morning sickness: `"first trimester nausea"`, `"hyperemesis gravidarum"`, `"LMP"`, `"gestational age"`

### 10.2 — Title tag formula

```
[Primary Keyword] — Free [Tool Type] | SageNest
```

Rules:
- Keep under 60 characters (SEOHead appends the site name)
- Primary keyword goes first — Google weights title beginnings more heavily
- Include a benefit word: Free, Personalized, Instant, No Signup
- Target the phrase people actually search, not a generic tool name

Good examples:
- `"When Does Morning Sickness End Calculator — Free Pregnancy Tool"`
- `"Pregnancy Flight Safety Calculator — Can I Fly Pregnant?"`
- `"Postpartum Hair Loss Timeline — Free Recovery Estimator"`

Bad examples:
- `"Morning Sickness Calculator"` — too generic, no benefit signal
- `"SageNest Nausea Tool"` — brand-first, keyword buried

### 10.3 — Meta description formula

```
[Action verb] your [primary keyword]. [What the tool outputs in one clause]. [Benefit/differentiator]. No signup required.
```

Rules:
- 140–160 characters
- Must contain the primary keyword
- Must describe the personalized OUTPUT the user gets (not just what the tool is)
- Must mention free/no signup

Good example:
`"Estimate when your morning sickness will end based on your LMP date. Get your personalized nausea peak and likely end date, based on ACOG guidelines. Free, no signup required."`

### 10.4 — H1 formula

```tsx
<h1 className="hero-title">[Condition/Topic] <span>[Calculator/Estimator/Checker]</span></h1>
```

The `<span>` makes the second part sage-primary green. The full H1 text must contain the primary keyword naturally.

Examples:
- `Morning Sickness <span>End Date Estimator</span>`
- `Pregnancy Flight <span>Safety Calculator</span>`
- `Postpartum Hair Loss <span>Timeline Calculator</span>`

### 10.5 — H2 structure for keyword coverage

The H2 headings in the educational section below the calculator are secondary keyword targets. Each H2 is an independent ranking opportunity for a different search query. Use this structure for every tool:

```
H2 #1: How does this [tool name] work?
        → captures "how does X work" informational queries

H2 #2: [Topic-specific heading containing secondary keyword]
        → your main secondary keyword target, written as a real question or statement

H2 #3: When should you talk to your doctor about [topic]?
        → captures safety/advice queries, strongly improves E-E-A-T signal

H2 #4: Frequently asked questions
        → captures all long-tail question queries via FAQ schema
```

Example for morning sickness tool:
- `How does this morning sickness end date estimator work?`
- `When does morning sickness typically end in pregnancy?`
- `When should you talk to your doctor about nausea in pregnancy?`
- `Frequently asked questions`

### 10.6 — FAQ questions as long-tail keyword slots

Each FAQ question must be written exactly as someone would type it into Google. These rank in "People Also Ask" boxes and as FAQ rich results via the FAQPage JSON-LD schema.

Rules:
- Complete sentence, exactly as a real person searches: `"Does morning sickness end at 12 weeks?"` not `"Morning sickness duration"`
- Include the core topic keyword in the question itself
- Minimum 5 questions, maximum 8
- Must cover these angles: how it works, accuracy/limitations, who this applies to, when to see a doctor, what the numbers mean
- Answers: 2–4 sentences — useful but scannable

### 10.7 — Trust badge as a keyword slot

The trust badge text is indexed by Google and appears above the H1. Use it to reinforce the primary keyword context:

Bad: `"Private, browser-only tool"` — generic, no keyword
Good: `"Free morning sickness end date estimate"` — keyword + benefit
Good: `"Based on ACOG clinical guidelines"` — authority signal
Good: `"Personalized pregnancy flight safety check"` — keyword + benefit

### 10.8 — Internal linking rules (mandatory)

Every tool page must link to at least two other pages on the site. This distributes authority and signals content relationships to Google.

**Required link 1 — Related tool, from results section:**
```tsx
import { Link } from 'react-router-dom';

<div style={{ marginTop: 'var(--space-md)' }}>
  <p style={{ marginBottom: 'var(--space-xs)' }}>Want your full pregnancy timeline?</p>
  <Link to="/pregnancy-due-date-calculator" style={{ color: 'var(--sage-primary)', fontWeight: 500 }}>
    Calculate your due date →
  </Link>
</div>
```

**Required link 2 — Related blog post, from educational section:**
```tsx
<p>
  Learn more in our guide to{' '}
  <a href="/blog/morning-sickness-remedies-that-actually-work" style={{ color: 'var(--sage-primary)' }}>
    morning sickness remedies that actually work
  </a>.
</p>
```

Use anchor text that contains a keyword — never `"click here"` or `"read more"`.

### 10.9 — Source citation as an E-E-A-T signal

Google's health content guidelines require visible sourcing. Cite the clinical source near the result or in the educational section — not buried in the footer.

```tsx
<p style={{ fontSize: '0.8125rem', color: 'var(--text-tertiary)', marginTop: 'var(--space-md)' }}>
  Estimates based on:{' '}
  <a
    href="[ACTUAL URL of the clinical source]"
    target="_blank"
    rel="noopener noreferrer"
    style={{ color: 'var(--text-tertiary)', textDecoration: 'underline' }}
  >
    [Full source name, e.g. ACOG Practice Bulletin #189, 2018]
  </a>
  . For informational use only.
</p>
```

Acceptable sources in order of E-E-A-T strength:
1. ACOG (acog.org)
2. NHS (nhs.uk)
3. AAD (aad.org)
4. CDC (cdc.gov)
5. WHO (who.int)
6. Named textbooks: Williams Obstetrics, etc.

Never cite: WebMD, BabyCenter, Healthline, or any secondary aggregator site.

### 10.10 — URL slug optimization

Rules:
- Must contain the primary keyword
- All lowercase, hyphen-separated, no trailing slash
- As short as possible while containing the keyword
- No stop words (a, the, for, of, in) unless part of the exact keyword phrase

Good: `/morning-sickness-end-date-calculator`
Bad: `/when-does-morning-sickness-end-calculator` — "when does" are stop words
Good: `/pregnancy-flight-calculator`
Bad: `/can-i-fly-while-pregnant-calculator` — too long and conversational
Good: `/postpartum-hair-loss-calculator`
Good: `/fundal-height-calculator`

## 12. What NOT to Do

- **Do not** add `import React from 'react'` — project uses new JSX transform
- **Do not** create a separate CSS file — use inline styles with design tokens or existing classes
- **Do not** add a `200` rewrite in `_redirects` for a tool page
- **Do not** touch `public/_headers`
- **Do not** use `localStorage` or `sessionStorage`
- **Do not** call any external API from tool code
- **Do not** use `export default` for page components — always named exports
- **Do not** edit `public/sitemap.xml` manually — auto-generated
- **Do not** change order of redirects in `_redirects` — catch-all must stay last
- **Do not** cite WebMD, BabyCenter, or any secondary health site as a source
- **Do not** write FAQ questions as topic phrases — write them as real complete-sentence search queries
- **Do not** use `"click here"` or `"read more"` as internal link anchor text

---

## 13. Route Naming Convention

Pattern: `/[condition-or-subject]-[calculator|estimator|checker|tracker]`

Slug must appear identically in: route, canonical URL, tools.ts path, _redirects, sitemap. All-lowercase, hyphen-separated, no trailing slash.
